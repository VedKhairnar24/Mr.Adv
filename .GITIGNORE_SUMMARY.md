**When you code:**
```bash
# Make changes to source files
git status               # See what changed

```

**Git automatically ignores:**
- node_modules/ (you don't commit dependencies)
- .env files (you don't commit secrets)
- dist/build/ (you don't commit builds)
- logs/ (you don't commit runtime files)

## 🎯 Best Practices

### DO ✅
- Commit source code only
- Use `.env.example` as template
- Keep `.gitignore` updated
- Review `git status` before committing
- Use environment variables for secrets

### DON'T ❌
- Commit `node_modules/`
- Commit `.env` with real secrets
- Commit `dist/` or `build/`
- Force add ignored files (`git add -f`)
- Hardcode passwords in code

---

## 🚨 If You Accidentally Commit Secrets

**Immediate Actions:**

1. **Rotate the secret:**
   - Change password/API key immediately
   - Update everywhere it's used

2. **Remove from Git history:**
   ```bash
   # Remove last commit (if just committed)
   git reset --soft HEAD~1
   
   # Or use BFG Repo-Cleaner for older commits
   ```

3. **Update .gitignore:**
   - Ensure the file type is ignored
   - Commit the .gitignore change

4. **Force push (carefully):**
   ```bash
   git push --force-with-lease
   ```

## 📞 Troubleshooting

### Q: "Why is Git still tracking node_modules?"

**A:** Check if you accidentally forced it:
```bash
# See if it's actually tracked
git ls-files | grep node_modules

# If yes, remove it
git rm -r --cached node_modules/
git commit -m "Remove node_modules"
```

### Q: "My .env file shows as untracked, is that okay?"

**A:** Yes! That's correct:
```bash
# It should be untracked (not committed)
git status
# Should show: .env under "Untracked files"
```

### Q: "How do I know what will be committed?"

**A:** Always check before committing:
```bash
# See staged changes
git status

# See what will be committed
git diff --staged

# Verify everything looks right
git commit -m "your message"
```
