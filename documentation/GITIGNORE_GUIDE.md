# Git Ignore Guide - Mr.Adv Advocate Case Management System

## 📋 Overview

This guide explains what files are ignored in each part of the project and why. The project uses **three levels** of `.gitignore` files:

1. **Root `.gitignore`** - Project-wide ignores
2. **`backend/.gitignore`** - Node.js backend specific ignores
3. **`frontend/.gitignore`** - React/Vite frontend specific ignores

---

## 🎯 What Gets Ignored (And Why)

### ❌ NEVER Commit These Files

#### 1. **Dependencies** (`node_modules/`)
- **Location:** `backend/node_modules/`, `frontend/node_modules/`
- **Why:** 
  - Massive folders (hundreds of MBs)
  - Easily regenerated with `npm install`
  - Different per OS/platform
  - Contains thousands of small files

#### 2. **Environment Variables** (`.env*`)
- **Files:** `.env`, `.env.local`, `.env.development.local`, etc.
- **Why:**
  - 🔒 **SECURITY RISK** - Contains secrets (database passwords, API keys, JWT secrets)
  - Different per developer/machine
  - Should never be public

#### 3. **Build Outputs** (`dist/`, `build/`)
- **Backend:** `dist/`, `build/`
- **Frontend:** `dist/`, `build/`, `dist-ssr/`
- **Why:**
  - Generated from source code
  - Can be recreated with `npm run build`
  - Large binary files
  - Changes with every deployment

#### 4. **Logs** (`*.log`, `logs/`)
- **Files:** `*.log`, `npm-debug.log*`, `yarn-error.log*`
- **Why:**
  - Runtime files that change constantly
  - Clutter Git history
  - Not useful for version control

#### 5. **Test Coverage** (`coverage/`)
- **Files:** `coverage/`, `.nyc_output/`, `jest-cache/`
- **Why:**
  - Generated by running tests
  - HTML reports are large
  - Can be regenerated anytime

#### 6. **IDE/Editor Files** (`.vscode/`, `.idea/`)
- **Files:** `.vscode/*`, `.idea/`, `*.swp`, `*.swo`
- **Why:**
  - Personal editor settings
  - Different per developer
  - Not relevant to code

#### 7. **OS Files** (`.DS_Store`, `Thumbs.db`)
- **Files:** `.DS_Store`, `Thumbs.db`, `Icon`, `._*`
- **Why:**
  - Automatically generated by operating system
  - Not relevant to code
  - Different per OS

#### 8. **Uploads** (`backend/uploads/*`)
- **Files:** User-uploaded documents and evidence
- **Why:**
  - User-generated content
  - Large files
  - Sensitive data
  - Different per installation

## 🔒 Security Best Practices

### Environment Variables

**NEVER commit these files:**
```bash
.env                           # Contains real secrets
.env.local                     # Local overrides
.env.development.local         # Dev-specific
.env.test.local                # Test-specific
.env.production.local          # Production-specific
```

**DO commit this template:**
```bash
.env.example                   # Safe template with placeholder values
```

**Example `.env.example`:**
```bash
# Database Configuration
DB_HOST=localhost
DB_PORT=3306
DB_NAME=advocate_case_db
DB_USER=root
DB_PASSWORD=your_password_here

# JWT Secret (change in production!)
JWT_SECRET=your_jwt_secret_here

# Server Configuration
PORT=5000
NODE_ENV=development
```

### What Makes It Dangerous?

**These should NEVER appear in Git:**
```javascript
// ❌ DANGEROUS - Hardcoded secrets
const DB_PASSWORD = "SuperSecret123!";
const JWT_SECRET = "my-secret-key";
const API_KEY = "sk-1234567890abcdef";

// ✅ SAFE - Use environment variables
const DB_PASSWORD = process.env.DB_PASSWORD;
const JWT_SECRET = process.env.JWT_SECRET;
const API_KEY = process.env.API_KEY;
```

## 📊 File Size Comparison

### What NOT to Commit (Large Files)

| Folder/File | Typical Size | Why Ignore |
|-------------|--------------|------------|
| `node_modules/` | 200-500 MB | Thousands of dependency packages |
| `dist/` or `build/` | 50-100 MB | Bundled application |
| `coverage/` | 10-20 MB | Test coverage HTML reports |
| `uploads/` | Variable | User files (can be huge) |
| `*.log` | Variable | Runtime logs (constantly changing) |
| `.vscode/` | Variable | Personal IDE settings |

### What TO Commit (Source Code)

| Folder/File | Typical Size | Why Commit |
|-------------|--------------|------------|
| `src/` | 1-5 MB | Your actual code |
| `config/` | < 1 MB | Configuration files |
| `package.json` | < 1 KB | Dependency list |
| `*.sql` | < 1 MB | Database schema |
| `*.md` | < 1 MB | Documentation |

## 🛠️ How to Use This System

### Initial Setup

1. **Clone the repository:**
   ```bash
   git clone <repository-url>
   cd Mr.Adv
   ```

2. **Setup Backend:**
   ```bash
   cd backend
   npm install
   cp .env.example .env  # Then edit with real values
   npm run dev
   ```

3. **Setup Frontend:**
   ```bash
   cd frontend
   npm install
   cp .env.example .env  # If needed
   npm run dev
   ```

### Adding New Files

**When you create new files, Git will automatically track them if they're not ignored.**

**To check what will be committed:**
```bash
git status          # See all changed/new files
git add .           # Stage all non-ignored files
git status          # Verify what's staged
```

**To verify ignores are working:**
```bash
git status --ignored    # Shows ignored files
git check-ignore -v <file>   # Check why a file is ignored
```

## 🚨 Common Mistakes to Avoid

### ❌ Mistake 1: Committing `.env` with Real Secrets

```bash
# WRONG - Don't do this!
git add .env
git commit -m "Add database config"
```

```bash
# RIGHT - Do this instead!
git add .env.example
git commit -m "Add environment template"
```

### ❌ Mistake 2: Forcing Addition of Ignored Files

```bash
# WRONG - Don't force add node_modules!
git add -f node_modules/
```

### ❌ Mistake 3: Removing `.gitignore` Rules

```bash
# WRONG - Don't remove ignore rules just to commit temporary files
# Edit .gitignore and remove: node_modules/
```

### ✅ Correct Workflow

```bash
# 1. Create your feature
# 2. Only source code changes are tracked
# 3. Verify with git status
git status
# 4. Commit your code (not dependencies/builds)
git add .
git commit -m "Add new feature"
```

## 🔍 Troubleshooting

### Issue: Accidentally Committed `node_modules/`

**Solution:**
```bash
# Remove from Git history (but keep locally)
git rm -r --cached node_modules/
git commit -m "Remove node_modules from tracking"

# Add to .gitignore (already done)
# Push changes
git push origin main
```

### Issue: `.env` File Showing as Untracked

**Solution:**
```bash
# Verify .gitignore includes .env
cat backend/.gitignore | grep ".env"

# Should show:
# .env
# .env.local
# .env.*.local
```

### Issue: Build Folder Being Tracked

**Solution:**
```bash
# Remove build/dist from Git
git rm -r --cached dist/
git rm -r --cached build/
git commit -m "Remove build outputs from tracking"
```

## 🎯 Best Practices

### 1. **Always Use `.env.example`**
Create a template file with placeholder values so other developers know what environment variables are needed.

### 2. **Never Commit Secrets**
If you accidentally commit secrets:
1. Rotate/change those secrets immediately
2. Remove from Git history
3. Update `.gitignore`

### 3. **Keep `.gitignore` Updated**
Add new ignore patterns as your project grows.

### 4. **Use Global `.gitignore`**
Set up a global `.gitignore` for OS/IDE files:
```bash
git config --global core.excludesfile ~/.gitignore_global
```

### 5. **Review Before Committing**
Always run `git status` before committing to ensure you're only committing what you intend to.
